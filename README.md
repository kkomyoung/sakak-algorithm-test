# 개미수열 가운데 두 자리

## 문제 설명

> 양의 정수 n이 주어질때  n번째 항(Ln)의 자릿수 중 가운데 두 자리 수(m)를 출력합니다.
(단, 첫 번째 항은 1이고, 입력 값 n은 3보다 크고 100보다 작습니다.)

개미수열(Look-and-Say Sequence)의 n번째 항에서 가운데 두 자리 숫자를 반환하는 문제입니다.

```
1항: 1
2항: 11 (1이 1개)
3항: 21 (1이 2개)
4항: 1211 (2가 1개, 1이 1개)
5항: 111221 (1이 1개, 2가 2개, 1이 2개)
```

## 실행 방법

```bash
# 의존성 설치
npm install

# 테스트 실행
npm test

# 개별 솔루션 실행 (디버깅용)
node src/solutions/solution1.js
node src/solutions/solution2.js
```

## 풀이 설명

### Solution 1: 기본 반복문

**접근 방식**
- 모든 항을 배열에 저장하며 순차적으로 계산
- 마지막에 문자열로 변환하여 중간 두 자리 추출

**문제점**
- 입력값 67부터 메모리 초과 발생
- 배열에 모든 값을 저장하여 불필요한 메모리 사용
- 수열의 길이가 눈덩이처럼 불어남


### Solution 2: 메모리 개선 버전

**접근 방식**
- 이전 항만 유지하고 새로운 항으로 교체하는 방식
- 문자열 변환 없이 배열 상태에서 중간 인덱스 계산

**개선한 부분**
- 배열에 모든 항을 누적하지 않도록 수정
- 기존 문자열 변환 후 숫자 변환 과정을 배열 상태에서 처리하도록 수정

**문제점**
- 여전히 입력값 69부터 메모리 초과 발생
- 항 자체의 길이가 엄청나게 증가하는 것은 피할 수 없음


## 결과

**실패:** n≥69 (메모리 부족)

- 실행 환경에 따라 결과는 다를 수 있지만 n=99 까지는 해당 코드로 해결 불가

## 소감

처음 문제를 받았을 때 별도의 코딩테스트 프로그램 없이 개인 환경에서 진행한다는 점이 의아했습니다.<br/>
검색을 통해 답을 찾을 수 있는 환경이기 때문입니다.<br/>
하지만 "다른 사람의 풀이법을 보지 않고 직접 문제를 풀어본 후 설명과 소감을 제출"이라는 요구사항을 보고<br/>
답을 아는 것 보다 문제를 해결하는 과정을 보고자 하는 것이라 생각했습니다.<br/>
그래서 막히더라도 일단 어떤 힌트도 보지 않고 풀어보려고 했습니다.

처음엔 일단 가장 직관적으로 떠오르는 방법으로 시작했습니다.<br/>
반복문을 사용해 문자열로 수열을 생성하는 방식이었고 작은 입력값에서는 테스트를 통과했습니다.<br/>
하지만 입력값이 커질수록 실행 속도가 느려지고 결국 메모리 부족 오류가 터졌습니다.

1차로 작성한 코드를 다시 분석하며 개선할 수 있는 포인트를 찾아보았습니다.
- 문자열로 변환하고 다시 숫자로 변환하는 불필요한 부분 제거
- 생성했던 모든 수열을 누적하여 저장하지 않고 필요한 이전 수열만 유지

이러한 개선으로 조금의 메모리 사용량을 줄일 수 있었지만 근본적인 문제는 해결되지 않았습니다.

그래서 모든 수를 구하지말고 내가 필요한 부분만 계산해보자라는 아이디어를 떠올렸습니다.<br/>
평소 방대한 양의 데이터를 화면에 렌더링할 때 가상 스크롤과 같은 최적화를 하지 않으면<br/>
앱이 튕겨버리는 문제가 있었던 것이 떠올랐고 비슷하게 접근해보기로 했습니다.<br/>
종이에 수열을 직접 써가며 반복되는 규칙을 찾으려 했으나 쉽게 규칙을 발견하지 못했습니다.

결국 시간 내에 해결이 불가능하겠다는 생각에 검색을 통해 힌트를 얻었습니다.
1. 내가 찾고자 하는 항의 전체 개수를 파악하여 중간 지점의 위치를 구한다
2. 연속되는 같은 숫자는 [개수, 숫자]로 구성된 1개의 묶음이자 숫자 2개로 바뀐다
3. 모든 정보를 알려고 하지 말고, 필요한 부분의 위치를 추적해서 답을 찾는다

하지만 직접 구현해보려고 시도하며 깨닫게 된 점은<br/>
결과적으로 위치를 알아도 정확한 중간 자리 2개 숫자를 구하기 위해서는<br/>
이전 항의 수 전체를 알아야 했다는 것입니다.

문제는 다시 원점으로 돌아왔습니다.<br/>
추가 검색을 통해 수열을 92개 원소로 분해하는 수학적 증명도 존재한다는 것을 알게 되었으나<br/>
실제 구현에는 상당한 수학적 배경과 시간이 필요할 것으로 보였습니다.

결국 완벽한 해결책을 시간 내에 찾지 못했지만 왜 안 되는지, 내가 작성한 코드의 문제점은 무엇인지,<br/>
여러 방법을 시도하며 한계를 분석하는 과정 자체가 유익했다고 생각합니다.
